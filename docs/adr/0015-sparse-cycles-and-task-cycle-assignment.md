# Sparse cycles and task cycle assignment

Church Task treats Weeks as Church Time Zone date math but persists Cycles sparsely: onboarding and rollover maintenance ensure the current and next Cycles exist, while farther future Weeks are rendered without rows until a user adds work, names/describes the Week, or records a Cycle Adjustment. Tasks may exist without a Cycle only while they remain To-do; active or completed Tasks must be attached to a Cycle, and Due Date never moves a Task between Cycles. Rollover maintenance moves every non-terminal Task from the closing Cycle into the next Cycle, leaves Done/Canceled Tasks behind, materializes that Week's Template Tasks idempotently with Cycle Adjustments applied, and creates the next Cycle if needed.

This replaces the earlier assumption that every Task always has a Cycle and that scheduled work pre-creates arbitrary future Cycles. The trade-off is deliberate: the UI can show future planning weeks without empty durable rows, while the domain still has real Cycles for active/current work, future user-authored planning, rollover, and Linear-style correction where a rolled-over Task can later be completed and moved back to the previous Cycle.
