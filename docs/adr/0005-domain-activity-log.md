# Domain activity log

Church Task will maintain a Church-scoped `activities` table for audit history across domain entities such as Tasks, Templates, Template Tasks, Cycles, Teams, and Workflows. Domain mutations that create, move, complete, materialize, roll over, cancel, reopen, or otherwise meaningfully change those entities must write corresponding Activity records in the same mutation transaction when possible, so UI and agent surfaces can show reliable history. Activities are an audit log, not event sourcing: current state remains on the domain tables, and the application must not require replaying Activities to reconstruct normal entity state.

Each Activity must carry enough structured detail to render what happened on its own, without re-deriving it from current state. The Activity Feed renders one human-readable line per Activity ("moved from To Do to In Progress", "assigned to Ankit", "set priority to High"), so a change that alters a value records the changed field as a dedicated `event_type` with `{ from, to }` in `metadata`. Each side stores both a stable id and a display `label` snapshot (`{ id, label }`); the Feed prefers the referenced record's current name when the id still resolves and falls back to the snapshot label when the record has been renamed away or deleted, so historical lines stay truthful even after Workflow Statuses, Labels, Users, or Teams change. The same structured before/after metadata also supports targeted restore flows such as reopening a canceled Task to its previous Workflow Status. A single mutation that changes several fields writes one Activity per meaningful field so the Feed shows them as separate lines. Purely mechanical or internal changes that are not worth surfacing — such as board ordering (`board_order`) drag reordering, and the automatic side effects of a Team change (Workflow remap, foreign Team Label drop, Task Identifier reissue) — are not recorded as their own Feed-facing Activities.
